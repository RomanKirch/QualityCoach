# ISO 25010 Quality Characteristics Reference

> These eight characteristics form the product quality dimension of the Holistic Quality Model (HQM).
> Copyright notice: Do not reproduce ISO/IEC 25010:2023 normative text verbatim.

## 1. Functional Suitability
Degree to which the product provides functions that meet stated and implied needs.
Sub-characteristics: Functional Completeness, Functional Correctness, Functional Appropriateness

When relevant: Almost always. Check completeness, edge cases, error paths, account states.
Typical gap: Requirements incomplete at the edges — error paths, empty states, role/permission boundaries.

Example criteria:
- Every distinct account state (active, locked, unverified, suspended) has an explicit criterion.
- Calculated values match the defined business rule in 100% of test cases.

## 2. Performance Efficiency
Performance relative to resources used under stated conditions.
Sub-characteristics: Time Behaviour, Resource Utilisation, Capacity

SLO patterns:
- p95 response time <= X ms under N concurrent users
- availability >= 99.9% per calendar month
- batch job completes within X hours for N records

When relevant: Any user-facing interaction, API, batch job, real-time feature, data-intensive op.

Example criteria:
- p95 API response time <= 300 ms under 500 concurrent users (load test).
- Batch completes within 4 h for 10 M records.

Verification: Load test (k6, Locust, Gatling) with p95/p99 assertions; APM latency alerts.

## 3. Compatibility
Exchange of information while sharing environment with other products.
Sub-characteristics: Co-existence, Interoperability

When relevant: Integration scenarios, API consumers, multi-tenant systems, microservices.

Example criteria:
- REST API responses conform to agreed OpenAPI spec, validated by contract test in CI.
- Consumer-driven contract tests prevent breaking changes from reaching production.

## 4. Usability
Specified users achieve specified goals with effectiveness, efficiency, and satisfaction.
Sub-characteristics: Appropriateness Recognisability, Learnability, Operability, User Error
Protection, User Interface Aesthetics, Accessibility

Key insight: Requires observing real users — reviewing designs is not sufficient.

Example criteria:
- New users complete onboarding in <= 5 min without help (usability test, n=5).
- All interactive elements keyboard-navigable with ARIA labels (WCAG 2.1 AA).

## 5. Reliability
Performs specified functions under specified conditions for a specified period.
Sub-characteristics: Maturity, Availability, Fault Tolerance, Recoverability

SLO patterns: availability >= 99.9%/month | RTO <= 1 h, RPO <= 15 min
Key insight: Design for failure. SLOs, error budgets, chaos engineering are the modern toolkit.

Example criteria:
- Service availability >= 99.9% per calendar month.
- Feature degrades gracefully when downstream service is unavailable (fallback, not 500).
- Automated retry with exponential backoff for transient async failures.

Verification: Chaos/fault injection, resilience testing, SLO monitoring with burn-rate alerts.

## 6. Security
Protects information so parties have the appropriate degree of access they are authorised for.
Sub-characteristics: Confidentiality, Integrity, Non-repudiation, Accountability, Authenticity

When relevant: Personal data (GDPR), financial/health data (HIPAA), auth/authz, public APIs.

Example criteria:
- All endpoints require valid token; unauthenticated requests return HTTP 401.
- User A cannot access User B data (IDOR check in security test suite).
- OWASP Top 10 scan passes with zero High/Critical findings before each release.

## 7. Maintainability
Effectiveness and efficiency with which a product can be modified.
Sub-characteristics: Modularity, Reusability, Analysability, Modifiability, Testability

Key insight: Poor maintainability is often a values/process gap as much as a technical one.

Example criteria:
- Any change in business rule X requires modifying only one module.
- Cyclomatic complexity <= 10 per function (enforced by linter in CI).

## 8. Portability
Effectiveness and efficiency of transferring from one environment to another.
Sub-characteristics: Adaptability, Installability, Replaceability

Example criteria:
- Docker image runs identically on AWS ECS, Azure AKS, and local Docker.
- Infrastructure described as IaC; deployment to new region is automated and reproducible.

## HQM Organisational Dimensions

Not part of ISO 25010, but central to HQM. Raise when they explain root causes of product quality gaps.

Process Quality: How work flows — release coordination, feedback loops, incident reviews.
People Quality: Knowledge, expertise, collaboration patterns, cross-functional engagement.
Values Quality: Priorities and trade-offs visible in everyday decisions — technical debt treatment,
incident learning, team empowerment to raise quality risks.
